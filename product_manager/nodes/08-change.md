# 08 - Change (变更分析)

## 步骤1 知识卡片加载
1. 识别当前用户输入内容中的核心话题/关键词
2. 读取知识卡片索引文件：product_manager/references/knowledge_cards/index.md
3. 根据识别出的话题匹配index中对应激活场景的知识卡片
4. 读取所有匹配成功的知识卡片完整内容，作为本节点后续输出的参考基础

## 步骤2 职责
- 评估变更影响，制定回退策略
- 变更类型判断
- PRD章节定位
- 版本复制与递增
- 增量变更执行

---

## 步骤3 变更影响分析

### 3.1 变更类型判断

根据用户提交的变更内容，分析变更类型并判断需要重新分析的节点：

| 变更类型 | 判断依据 | 回退节点 |
|---------|---------|---------|
| **想法/新方向** | 用户提交的是一个想法 | brainstorm |
| **功能（涉及业务流程调整或需提交文档）** | 用户提交的是功能，涉业务流程调整 | clarify |
| **功能（涉及新页面和业务）** | 用户提交功能，涉新页面和流程调整 | analysis |
| **文案/布局调整** | 仅涉及文案调整或页面布局 | detail |

### 3.2 PRD章节定位规则

**按大标题定位节点产出**：根据变更涉及的节点，在对应大标题下更新产出内容

| PRD大标题 | 负责节点 | 说明 |
|-----------|---------|------|
| 项目概述 | brainstorm/clarify | 需求背景和目标 |
| 需求分析 | analysis | 业务流程和页面列表 |
| 详细设计 | detail | 事件列表和异常场景 |
| 页面详细设计 | detail | 按页面维度组织详细内容 |
| 原型与交互 | genHTML | 原型引用和交互说明 |

**按页面名称定位具体内容**：在节点产出内按页面名称查找需要更新的页面和事件

### 3.3 版本号管理规则

- **涉及前序内容修改**：major+1（V1.x → V2.0）
- **仅为新增功能**：minor+1（V1.0 → V1.1）

---

## 步骤4 变更执行流程

### 4.1 接收用户变更内容

1. 接收用户描述的变更内容
2. 分析变更属于哪种类型（想法/功能/文案布局）
3. 判断需要回退的节点

### 4.2 变更影响分析

1. 读取当前版本PRD文档
2. 按大标题定位需要更新的章节
3. 按页面名称定位需要更新的具体内容
4. 展示变更影响分析报告

```
## 变更影响分析报告

### 变更类型
- 类型：[想法/功能/文案布局]
- 说明：[变更内容的简要描述]

### 影响章节
- [ ] PRD文档章节1
- [ ] PRD文档章节2

### 建议回退节点
- 回退到：[节点名称]
- 理由：[判断依据]

### 版本策略
- 版本递增：[major/minor]
- 新版本号：[Vx.x → Vx.x]
```

### 4.3 版本复制

确认变更影响分析后，执行版本复制：

1. 复制当前PRD文档为新版本文件：`PRD_V{x+1}.md`
2. 复制当前prototype目录为新版本目录
3. 在新版本PRD文档中标记变更记录

### 4.4 回退执行

1. 调用 manage.transition_to_first_affected_node(受影响节点列表)
2. 更新Memory.json的版本号和文件路径
3. 回退到第一个受影响的节点重新执行

---

## 步骤5 完成后

1. 展示产出摘要
2. 等待用户确认
3. 确认后 → 执行版本复制（PRD文档和prototype目录）
4. 确认后 → 更新Memory.json的版本号和文件路径
5. 确认后 → 调用 manage.update_node_status("change", "CONFIRM")
6. 确认后 → 调用 manage.transition_to_first_affected_node(受影响节点列表)
7. 确认后 → 调用 manage.persist_memory()
8. 回退到第一个受影响的节点
