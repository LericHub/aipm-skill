# 需求分析方法论与框架

## 一、需求分析五步法

### 1. 场景还原
**目的**: 理解用户在什么场景下遇到问题

**方法**:
- 用户访谈
- 实地观察
- 日志分析
- 问卷调查

**输出**:
- 用户场景描述
- 使用环境分析
- 触发条件梳理

### 2. 痛点挖掘
**目的**: 找到问题的根本原因

**工具**:
- 5Why分析法
- 鱼骨图
- 用户旅程地图

**示例**:
```
问题: 用户投诉搜索功能不好用
Why 1: 搜索结果不准确
Why 2: 搜索算法没有考虑用户意图
Why 3: 缺少用户行为数据分析
Why 4: 没有埋点收集用户搜索行为
Why 5: 产品规划时未考虑数据驱动

根因: 产品设计阶段缺乏数据思维
```

### 3. 价值评估
**目的**: 判断解决问题的价值大小

**评估维度**:
| 维度 | 问题 |
|------|------|
| 用户价值 | 解决了多少用户的痛点？ |
| 业务价值 | 对业务指标有什么影响？ |
| 战略价值 | 是否符合产品战略方向？ |
| 竞争价值 | 能否形成差异化优势？ |

### 4. 方案设计
**目的**: 提出可行的解决方案

**设计原则**:
- MVP优先: 先做最小可行方案
- 用户中心: 以用户体验为导向
- 技术可行: 考虑实现难度
- 成本可控: 评估资源投入

### 5. 优先排序
**目的**: 确定实施顺序

**排序方法**:

#### Kano模型
| 类型 | 定义 | 策略 |
|------|------|------|
| 基本型 | 必须有，没有会不满 | 优先保证 |
| 期望型 | 越多越好 | 按需投入 |
| 魅力型 | 意外惊喜 | 适度投入 |
| 无差异 | 有无都行 | 不投入 |
| 反向型 | 越多越不满 | 避免 |

#### MoSCoW法则
| 类型 | 定义 | 占比建议 |
|------|------|----------|
| Must have | 必须有 | 60% |
| Should have | 应该有 | 20% |
| Could have | 可以有 | 15% |
| Won't have | 暂不做 | 5% |

## 二、需求优先级评估矩阵

### ICE评分法

```
ICE得分 = Impact(影响) × Confidence(信心) ÷ Effort(投入)
```

| 维度 | 评分标准 (1-10) |
|------|-----------------|
| Impact | 1=影响很小，10=影响巨大 |
| Confidence | 1=不确定，10=非常确定 |
| Effort | 1=投入很小，10=投入巨大 |

### RICE评分法

```
RICE得分 = Reach(覆盖) × Impact(影响) × Confidence(信心) ÷ Effort(投入)
```

## 三、用户故事拆分技巧

### 按工作流拆分
```
Epic: 订单管理
├── 创建订单
├── 支付订单
├── 查看订单
├── 取消订单
└── 评价订单
```

### 按规则拆分
```
Epic: 用户注册
├── 基础注册（邮箱）
├── 手机号注册
├── 第三方登录
└── 邀请码注册
```

### 按界面拆分
```
Epic: 商品展示
├── 商品列表页
├── 商品详情页
├── 商品搜索
└── 商品筛选
```

### 按数据拆分
```
Epic: 数据报表
├── 日报表
├── 周报表
├── 月报表
└── 自定义报表
```
