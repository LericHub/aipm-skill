# Detail 节点事件描述交付标准优化计划

## 背景

当前 `06-detail.md` 节点中关于页面事件的描述规范存在以下问题，导致交付给开发时理解成本较高：

1. **事件定义粒度模糊**：同一事件内混杂了前端交互、接口调用、状态变更，开发难以判断哪些需要前端处理、哪些需要后端支持。
2. **缺少前端/后端职责边界**：事件逻辑中没有明确标注接口调用方、数据消费方。
3. **前置判断条件缺失**：没有说明事件触发前需要满足什么条件（如登录状态、权限、数据就绪状态）。
4. **数据来源与去向不清晰**：没有追溯数据从哪来、处理后到哪去（落库/下游/展示）。
5. **异常场景覆盖不全**：异常编号与事件编号关联不紧密，缺少前端异常（如网络超时、渲染失败）的明确定义。
6. **缺少状态机描述**：复杂事件（如兑换流程）涉及多状态流转，当前模板没有要求描述状态变更规则。
7. **埋点与事件耦合度低**：埋点方案独立于事件列表，开发需要二次映射。

## 目标

优化 `06-detail.md` 中「页面结构和事件定义」章节的输出模板，使其达到可直接交付给前端/后端开发执行的标准。

## 完整事件描述维度框架

一个页面上完整的事件描述应包含以下 **9 大维度**、**25 个字段**：

### 维度1：事件身份（我是谁）

| 字段 | 说明 | 示例 |
|------|------|------|
| 事件编号 | 全局唯一标识 | `/home/feat-01` |
| 事件名称 | 中文名称，清晰表达事件意图 | 查询会员信息 |
| 事件类型 | 枚举：页面加载 / 用户点击 / 数据变更 / 系统回调 / 定时触发 | 页面加载 |
| 关联用户故事 | 追溯需求来源 | US-001 |

### 维度2：触发机制（什么时候发生）

| 字段 | 说明 | 示例 |
|------|------|------|
| 触发节点 | 用户操作或系统时机 | 页面 `onLoad` 生命周期 |
| 触发条件 | **前置判断条件**，精确描述触发前必须满足的状态 | 用户已登录（token 有效）且非静默登录态 |
| 防抖/节流策略 | 是否需要防重复触发 | 无需防抖，但需 Loading 锁防止并发请求 |

### 维度3：数据输入（需要什么数据）

| 字段 | 说明 | 示例 |
|------|------|------|
| 数据来源 | 数据从哪来：接口 / 缓存 / 路由参数 / 全局状态 / 用户输入 | 接口：会员中台 `/api/member/query` |
| 入参契约 | 请求参数字段、类型、必填、示例值 | 见数据契约表格 |
| 前置数据依赖 | 触发本事件前，必须先完成哪些事件或数据准备 | 依赖 `/home/feat-00` 登录态初始化完成 |

### 维度4：职责边界（谁做什么）

| 字段 | 说明 | 示例 |
|------|------|------|
| 前端职责 | 前端需要完成的操作 | 1. 调用接口 2. 更新 State 3. 渲染积分和等级 4. 异常时展示兜底 UI |
| 后端职责 | 后端需要提供的支持 | 提供会员信息查询接口，返回积分、等级、有效期 |
| 第三方依赖 | 是否依赖外部系统或中台 | 依赖会员信息中台 3.2.2 接口 |

### 维度5：状态变更（变成什么）

> 注：状态机由统一章节维护（其他 session 处理），此处只说明事件执行后的**目标状态**。

| 字段 | 说明 | 示例 |
|------|------|------|
| 目标状态 | 事件执行后系统状态变更为 | 兑换订单状态 → "已提交" |
| 状态对应 UI | 该状态下的界面表现 | 展示"兑换成功，正在发货" |

### 维度6：数据输出（数据去哪）

| 字段 | 说明 | 示例 |
|------|------|------|
| 数据去向 | **数据处理后到哪去**：落库 / 下游接口 / 页面展示 / 全局状态 / 本地缓存 | 1. 页面展示（积分余额） 2. 写入全局状态 `userInfo` 3. 本地缓存（有效期 5 分钟） |
| 出参契约 | 响应数据字段、类型、映射位置 | 见响应契约表格 |
| 副作用 | 事件触发后对其他模块或数据的影响 | 更新全局用户状态，触发 `/home/feat-02` 商品列表加载 |

### 维度7：安全与权限（谁能做）

| 字段 | 说明 | 示例 |
|------|------|------|
| 权限要求 | 执行事件需要的权限 | 需登录且为品牌会员 |
| 数据权限 | 数据可见范围 | 仅可见当前店铺数据 |
| 幂等性 | 重复触发是否安全 | 查询类接口天然幂等，写入类需防重 |

### 维度8：异常处理（出错怎么办）

| 字段 | 说明 | 示例 |
|------|------|------|
| 异常场景 | 可能的错误情况 | 接口超时、会员不存在、数据格式错误 |
| 异常编号 | 与事件编号关联的异常标识 | `/home/feat-01/err-01` |
| 异常分类 | 网络异常 / 业务异常 / 系统异常 / 前端异常 | 网络异常 |
| 判断方式 | 如何识别该异常 | HTTP 状态码 != 200 或请求超时 > 10s |
| 处理方案 | 前端如何处理 | 展示"网络异常，请检查网络后重试"，提供重试按钮 |
| 用户体验 | 用户感知和恢复路径 | 用户可感知问题并自助恢复 |

### 维度9：埋点与监控（怎么追踪）

| 字段 | 说明 | 示例 |
|------|------|------|
| 关联埋点 | 事件触发的埋点编号 | TRACK-001 |
| 埋点触发时机 | 埋点在事件流中的精确触发点 | 接口返回成功后触发，非页面加载时 |
| 性能监控 | 是否需要监控耗时或成功率 | 需监控接口耗时，> 3s 上报慢请求 |

## 具体实施步骤

### 步骤1：重构事件列表模板（事件基础信息）

**当前问题**：事件描述过于笼统，缺少触发条件细节和职责划分。

**优化方案**：在事件列表中增加以下字段：

| 新增字段 | 说明 |
|---------|------|
| `事件类型` | 枚举：页面加载 / 用户点击 / 数据变更 / 系统回调 / 定时触发 |
| `触发条件` | **前置判断条件**，精确描述触发时机（如：页面 `onLoad` 生命周期 / 点击按钮且按钮状态为可用 / 接收到 WebSocket 推送） |
| `数据来源` | 数据从哪来：接口 / 缓存 / 路由参数 / 全局状态 / 用户输入 |
| `前端职责` | 前端需要完成的操作（如：调用接口、更新 State、跳转页面、弹窗提示） |
| `后端职责` | 后端需要提供的支持（如：提供接口、推送消息、定时任务） |
| `数据去向` | **数据处理后到哪去**：落库 / 下游接口 / 页面展示 / 全局状态 / 本地缓存 |
| `关联埋点` | 直接关联埋点编号（如 TRACK-001），建立事件与埋点的映射 |

**示例对比**：

- **优化前**：
  | 事件编号 | 事件名称 | 绑定页面 | 触发节点 | 事件描述 | 关联用户故事 |
  | /home/feat-01 | 查询会员信息 | 首页 | 载入时 | 调用会员中台接口查询用户积分和等级信息 | US-001 |

- **优化后**：
  | 事件编号 | 事件名称 | 事件类型 | 绑定页面 | 触发条件 | 数据来源 | 前端职责 | 后端职责 | 数据去向 | 关联埋点 | 关联用户故事 |
  | /home/feat-01 | 查询会员信息 | 页面加载 | 首页 | 页面 `onLoad` 且登录态有效 | 会员中台接口 `/api/member/query` | 1. 调用接口 2. 渲染积分和等级 3. 异常时兜底 | 提供会员信息查询接口 | 1. 页面展示 2. 全局状态 3. 本地缓存 | TRACK-001 | US-001 |

### 步骤2：重构事件逻辑模板（事件详细逻辑）

**当前问题**：事件逻辑分散为4个独立部分（流程图、页面流程、接口逻辑、异常），缺少统一的状态视角和数据血缘。

**优化方案**：将事件逻辑整合为「**前置条件 → 数据输入 → 目标状态 → 数据输出 → 异常处理**」五段式描述：

#### 2.1 增加「前置条件」章节

```markdown
##### {事件编号} 前置条件

| 条件项 | 判断规则 | 不满足时的处理 |
|--------|---------|---------------|
| 登录态校验 | `token` 存在且未过期 | 跳转登录页 |
| 权限校验 | 用户有当前店铺数据权限 | 展示无权限提示 |
| 数据依赖 | `/home/feat-00` 已完成 | 等待依赖事件完成或超时提示 |
```

#### 2.2 增加「数据血缘」章节

```markdown
##### {事件编号} 数据血缘

**输入数据**：
| 字段 | 来源 | 传递方式 | 说明 |
|------|------|---------|------|
| brand_code | 全局配置 | 注入请求头 | 当前品牌编码 |
| openId | 登录态 | 全局状态 | 用户唯一标识 |

**输出数据**：
| 字段 | 去向 | 消费方 | 说明 |
|------|------|--------|------|
| points | 页面展示 | 首页积分组件 | 展示当前积分余额 |
| points | 全局状态 | `userInfo` Store | 供其他页面读取 |
| points | 本地缓存 | `localStorage` | 有效期 5 分钟，减少重复请求 |
| level | 页面展示 | 首页等级标识 | 展示会员等级图标 |
```

#### 2.3 增加「目标状态」说明

> 注：状态机流转由统一章节维护，此处只说明事件执行后的目标状态。

```markdown
##### {事件编号} 目标状态

| 字段 | 值 | 说明 |
|------|-----|------|
| 订单状态 | "已提交" | 用户提交兑换后，订单状态变更为已提交 |
| 积分状态 | "扣减中" | 调用积分扣减接口后，积分状态变更为扣减中 |
```

#### 2.4 优化「接口交互逻辑」为「数据契约」

将接口逻辑从描述性文字改为结构化表格，明确数据契约：

```markdown
**请求契约**

| 字段 | 类型 | 必填 | 来源 | 说明 | 示例值 |
|------|------|------|------|------|--------|
| brand_code | string | 是 | 全局配置 | 品牌编码 | "LOREAL" |
| queryType | string | 是 | 硬编码 | 查询类型 | "OPENID" |
| value | string | 是 | 全局状态 | 用户 openId | "o123456789" |

**响应契约**

| 字段 | 类型 | 说明 | 映射 UI 位置 | 数据去向 |
|------|------|------|-------------|---------|
| data.points | number | 当前积分余额 | 首页积分展示区域 | 页面展示 + 全局状态 + 本地缓存 |
| data.level | string | 会员等级 | 首页等级标识 | 页面展示 |
| data.expireDate | string | 积分过期时间 | 首页积分过期提示 | 页面展示 |
```

### 步骤3：重构异常处理模板

**当前问题**：异常编号与事件编号关联不紧密（如 `/home-feat-01-err01` 与 `/home/feat-01` 格式不统一），缺少前端异常分类。

**优化方案**：

1. **统一异常编号规范**：`{事件编号}/err-{序号}`（与当前模板一致，但强制要求与事件编号关联）
2. **增加异常分类**：

| 异常分类 | 说明 | 处理方 |
|---------|------|--------|
| 网络异常 | 请求超时、断网、DNS 错误 | 前端 |
| 业务异常 | 接口返回错误码（如积分不足、库存不足） | 前端展示 + 后端提供错误码 |
| 系统异常 | 接口 500、服务不可用 | 前端兜底 + 后端告警 |
| 前端异常 | 渲染错误、JS 报错、数据格式不匹配 | 前端捕获 + 监控上报 |

3. **异常处理模板增加「用户体验」列**：

```markdown
| 异常场景 | 异常编号 | 异常分类 | 判断方式 | 前端处理方案 | 用户体验 |
|---------|---------|---------|---------|-------------|---------|
| 接口调用失败 | /home/feat-01/err-01 | 网络异常 | HTTP 状态码 != 200 或请求超时 | 展示"网络异常，请检查网络后重试"，提供重试按钮 | 用户可感知问题并自助恢复 |
| 会员信息不存在 | /home/feat-01/err-02 | 业务异常 | 返回 code = "MEMBER_NOT_FOUND" | 提示"您还不是本品牌会员"，引导跳转注册页 | 明确引导用户下一步操作 |
| 数据格式错误 | /home/feat-01/err-03 | 前端异常 | 积分字段为空或非数字 | 展示积分为 0，上报监控，提示"数据加载异常" | 降级展示，不阻断主流程 |
```

### 步骤4：建立事件与埋点的强关联

**当前问题**：埋点方案独立在第6章，开发需要手动建立事件与埋点的映射关系。

**优化方案**：

1. 在事件列表中增加「关联埋点」字段（见步骤1）
2. 在事件逻辑中增加「埋点触发时机」子章节：

```markdown
##### {事件编号} 埋点触发时机

| 埋点编号 | 埋点名称 | 触发时机 | 采集字段 | 上报条件 |
|---------|---------|---------|---------|---------|
| TRACK-001 | 页面曝光-首页 | 页面 onLoad 且数据加载完成 | 用户ID、活动ID、访问时间 | 每次进入首页触发 |
| TRACK-004 | 行为-商品点击 | 用户点击商品卡片 | 用户ID、商品ID、点击位置 | 点击即触发，不等待接口返回 |
```

### 步骤5：增加「前端状态管理」建议

**新增章节**：对于复杂页面，给出前端状态管理的建议：

```markdown
##### {页面名称} 状态管理建议

| 状态域 | 状态说明 | 初始值 | 更新时机 | 影响范围 |
|--------|---------|--------|---------|---------|
| userInfo | 用户信息（积分、等级） | null | 页面加载完成后 | 全局 |
| productList | 商品列表数据 | [] | 接口返回后 | 当前页面 |
| loadingStatus | 加载状态 | "idle" | 请求发起/返回/异常时 | 当前页面 |
| errorInfo | 错误信息 | null | 异常发生时 | 当前页面（Toast/弹窗） |
```

## 预期收益

1. **降低沟通成本**：开发无需反复确认事件边界、接口契约、数据流向
2. **提升开发效率**：数据契约、数据血缘可直接转化为代码逻辑
3. **减少遗漏**：异常分类、前置条件、数据去向确保场景覆盖完整
4. **可追溯性**：事件编号与异常编号、埋点编号、数据血缘强关联，便于后续维护和排查

## 风险与回退方案

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 模板过于复杂，导致输出内容膨胀 | 节点执行时间增加，文档可读性下降 | 将「数据血缘」设为可选章节，仅复杂事件需要 |
| 与现有 Penpot 绘制逻辑不兼容 | 绘制内容超出单栏宽度 | Penpot 栏宽从 420px 调整为 480px，或复杂内容单独一栏 |
| 开发反馈字段过多 | 实际使用中部分字段被忽略 | 提供「精简模式」和「完整模式」两种输出选项 |
