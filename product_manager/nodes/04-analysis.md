# 04 - Analysis (需求分析)

> **注意**：知识卡片加载已由 Router 统一处理。本节点接收 Router 传递的知识卡片内容作为输入。

## 步骤1 职责
- 构建 PRD 骨架，识别用户角色、页面清单
- 页面编号生成规则：`/{pageNameInEN}`（如 `/home`、`/productList`、`/productDetail`）
- 事件编号生成规则：`/{pageNameInEN}/feat-{序号}`
- 异常场景编号规则：`/{pageNameInEN}/feat-{序号}/err-{序号}`

---

## 步骤2 加载指引

### 前序节点产出物

从 Memory.json 获取 clarify 的 `output.path`：
- 调用 manage.read_memory() 获取 node_list
- 从 node_list 中获取 clarify 的 `output.path` 字段
- 读取该文件内容

---

## 步骤3 执行逻辑

1. 解析 clarify 产出的文件内容
2. 按照输出模板生成 PRD，将内容直接输出到用户消息中（如果包含 Mermaid 代码块，需要特殊说明以便渲染）
3. **生成后立即写入文件**（创建目录 + 保存 PRD + 更新 Memory.json，详见 5.2）
4. 等待用户确认交付物
5. 若用户要求修改 → 修改内容后重新写入文件（详见 5.5）→ 再次等待确认
6. 用户最终确认后 → 流转到下一节点（详见步骤7）

---

## 步骤4 需求分析与交付

**🚨 关键规则：初步的 PRD 也必须非常详细，不能只是一个空框架。**

### 4.1 产出内容

根据澄清后的确认信息，直接生成一份详细的初步 **Markdown 格式**文档（同时将文档内容以文本方式输出给用户）。这份初版 PRD 必须搭建好完整的文档结构骨架，并且必须深度填充以下部分：

1. **项目基本信息**
2. **需求背景与目标**：先说明需求背景，然后同列表的方式展示业务目标。业务目标必须包含四列表格（目标类型、描述、衡量指标、目标值）。
3. **识别用户类型和端口**：
    - 识别用户类型和端口。
    - 根据用户类型和端口，梳理用户在使用产品时的场景和核心流程。（先使用文字描述流程节点，如用户下单-》订单确认-》订单支付）
4. **识别业务对象**：用 Markdown 表格输出业务对象和关键属性（如用户、订单、商品等），表格格式如下：

```markdown
| 业务对象 | 对象属性 |
|---------|---------|
| 用户 | 用户ID、用户名、手机号、邮箱、注册时间、最后登录时间、用户等级、积分余额 |
| 订单 | 订单ID、用户ID、订单状态、订单金额、支付方式、下单时间、支付时间 |
| 商品 | 商品ID、商品名称、商品描述、商品价格、库存数量、兑换所需积分、上架状态 |
```

**🚨 必须使用上述表格格式！**

5. **业务流程图**：
  - 用 Mermaid 图表输出业务流程图，展示用户在使用产品时的场景和核心流程。
  - 必须使用横向的流程图

6. **拆解用户故事**：将业务需求拆解成多个用户故事
```markdown
| 故事编号 | 端口 | 用户故事 |
|---------|---------|---------|
| S01 | 小程序端 | 作为一个消费者，我需要浏览商品列表、查看商品详情、下单、支付、查看订单详情 |
| S02 | 管理端 | 作为一个管理员，我需要配置活动的参数 |
```
7. **页面列表**：根据用户故事整理页面列表
  - 用列表输出B、C端页面列表，需要按照页面功能和用户操作两个维度描述。
8. **埋点方案**：必须使用表格输出埋点场景的埋点的字段。
9. **异常场景**：对接系统返回的异常状态处理。（接口调用异常、接口返回业务异常）
*（后续章节如详细设计、页面详细设计、原型与交互等，在对应节点完成后自动追加到PRD中，当前analysis节点不生成这些章节）*

### 4.2 写入文件（生成后立即执行）

> **🚨 关键规则：PRD 内容生成后必须立即写入文件，不等待用户确认。如果用户后续要求修改，修改后重新写文件。**

1. 创建版本目录：`./{项目名称}_{datetime}/output/V1.0/`
2. 读取 `analysis_process/clarify.md` 内容（brainstorm和clarify整合内容）
3. 生成 PRD 文档框架，将 brainstorm/clarify 内容填入"项目概述"章节
4. 将 analysis 节点产出填入"需求分析"章节
5. 保存产出到文件：`./{项目名称}_{datetime}/output/V1.0/PRD_V1.0.md`
6. 调用 `manage.update_node_output("analysis", {"type": "prd", "path": "{项目名称}_{datetime}/output/V1.0/PRD_V1.0.md"})`
7. 展示产出摘要

### 4.3 用户确认 【等待用户确认】

询问用户：这是产品的详细第一版架构和业务逻辑，您看方向和基础逻辑准确吗？如果没问题，我们将先进行原型设计，通过具体的画面来进一步理清交互细节和可能遗漏的功能。

**🛑 强制暂停：在此处必须停止输出！等待用户回复确认后，进入步骤7完成流转。**

### 4.4 用户验收标准  

产出展示后，明确告知用户以下验收规则：

**验收通过意图（可进入下一节点）：**
- 明确确认："确认"、"没问题"、"可以"、"好的"、"通过"、"验收通过"
- 推进指令："继续"、"下一步"、"往下走"、"进入下一阶段"
- 满意表达："满意"、"就这样"、"定稿"、"OK"、"ok"

**非验收意图（停留在当前节点）：**
- 修改请求："修改"、"不对"、"重来"、"重新生成"、"调整"、"改一下" → 进入 5.5 修改流程
- 补充需求："补充"、"还有"、"另外"、"加上"、"漏了" → 补充内容后重新展示
- 疑问咨询："为什么"、"什么意思"、"不懂"、"解释一下" → 解答疑问后继续等待确认
- 部分确认："这部分可以"、"其他的再改改" → 视为未完全确认，继续修改
- 拖延/犹豫："再看看"、"先这样"、"待定" → 明确询问用户是否确认

**重要规则：**
- **只有用户明确表达验收通过意图时，才能推进到 drawUserStory 节点**
- **任何非验收意图（修改、补充、疑问、部分确认、犹豫）都必须停留在 analysis 节点，不得进入下一节点**
- 用户表达"变更"、"加功能"、"改需求"等意图时，进入 change 节点

### 4.5 用户要求修改时

若用户提出修改请求：

1. 根据用户反馈修改 PRD 内容
2. **重新写入文件**（覆盖 4.2 中创建的文件）
3. 更新 Memory.json 中的 `output.path`
4. 展示修改后的产出摘要
5. 回到 4.3 等待用户再次确认

## 步骤5 输出文件

### 5.1 目录初始化

在analysis节点完成时，创建版本目录并生成PRD文档框架：

```
./{项目名称}_{datetime}/output/V1.0/
├── PRD_V1.0.md           # PRD文档（初始版本）
└── prototype/            # 原型目录（在genHTML节点创建）
```

### 5.2 PRD文档框架模板

生成PRD文档，框架包含以下章节。

**🚨 关键规则：只包含当前已生成内容的章节，未生成内容的章节不出现在PRD中。**

- analysis节点首次生成时，只包含第1、2章节
- detail节点完成后，追加第3、4章节
- genHTML节点完成后，追加第5章节
- change节点维护第6章节

**analysis节点首次生成的PRD框架：**

```markdown
# PRD文档 - V1.0

## 1. 项目概述
### 1.1 需求背景与目标 [brainstorm/clarify节点产出]
### 1.2 用户角色

## 2. 需求分析
### 2.1 业务流程图 [analysis节点产出]
### 2.2 页面列表
### 2.3 功能列表
```

**完整PRD框架（所有节点完成后）：**

```markdown

## 2. 需求分析
### 2.1 业务流程图 [analysis节点产出]

## 4. 页面详细设计

#### 4.1.5 异常场景

## 5. 原型与交互
```

### 5.3 输出路径

- 文件名: `PRD_{Version}.md`
- 路径: `./{项目名称}_{datetime}/output/{Version}/PRD_{Version}.md`
- 生成规则: 见 [01-router.md](01-router.md) 中的文件生成规则
