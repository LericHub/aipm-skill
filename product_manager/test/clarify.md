# 需求澄清

## 一、背景与痛点

> 已由 BRAINSTORM 阶段确认
>
> 搭建一个部署在天猫小程序端的会员兑礼小程序，纯积分兑礼。这是一个从0-1搭建的应用。

**归纳扩展：**
- **业务场景**：天猫商家需要为会员提供积分兑换礼品的能力，目前缺少独立的小程序端兑礼入口
- **核心痛点**：
  - 会员积分消耗渠道有限，缺乏积分价值感知
  - 线下兑礼流程繁琐，需要线上化提升体验
  - 缺少统一的兑礼活动管理平台，运营效率低

---

## 二、业务目标

> 已由 BRAINSTORM 阶段确认
>
> 会员小程序端提供兑礼能力，B端管理端提供活动管理能力。

**归纳扩展：**
- **业务目标**：
  - 为会员提供便捷的积分兑礼入口，提升积分价值感知
  - 为运营提供多渠道、多店铺的活动管理能力
  - 实现兑礼流程线上化、自动化，降低人工成本
- **成功指标**：
  - 小程序端会员兑礼参与率
  - 积分消耗转化率
  - 兑礼订单完成率

## 五、业务规则

### 5.1 兑礼规则

| 规则项 | 规则说明 |
|-------|---------|
| 兑礼方式 | 纯积分兑礼，不支持混合支付 |
| 用户兑礼控制 | 由中台控制，无特殊业务规则 |
| 商品库存 | 待确认（由兑礼中台提供） |
| 地址限制 | 淘宝地址列表，无地区限制 |

### 5.2 订单状态流转

```
正向流程：
用户兑礼 → 积分扣减 → 订单记录 → 订单回传 → 订单状态="已完成"

逆向流程：
用户兑礼 → 积分扣减 → 扣减失败 → 订单状态="已取消"
```

### 5.3 订单状态定义

| 状态 | 说明 |
|------|------|
| 已完成 | 积分扣减成功，订单回传成功 |
| 已取消 | 积分扣减失败 |

## 六、业务风险

### 6.1 已识别风险

| 风险场景 | 风险描述 | 处理方案 |
|---------|---------|---------|
| 积分扣减失败 | 用户积分不足或中台异常 | 页面显示Toast"兑换失败，积分扣减失败"，订单状态标记为"已取消" |
| 订单同步失败 | 订单回传中台失败 | 钉钉预警通知，人工运维重推 |

### 6.2 风险处理流程

**积分扣减失败流程**：
1. 用户点击兑礼
2. 调用积分扣减接口失败
3. 页面提示："兑换失败，积分扣减失败"
4. 订单状态自动标记为"已取消"
5. 用户可重新尝试兑礼

**订单同步失败流程**：
1. 兑礼成功后，订单回传中台失败
2. 触发钉钉预警通知
3. 运维人员手动重推订单
