# 为所有节点增加完整验收意图判定规则

## 背景

当前各节点在输出环节等待用户确认时，缺乏明确的验收标准说明。用户不清楚需要回复什么意图才算验收通过，导致节点流转不清晰。

## 目标

为每个业务节点（02-brainstorm 至 09-change）增加完整的验收意图判定规则，确保：
- **只有用户明确表达验收通过的意图时，才能推进到下一个节点**
- **用户表达修改、补充、疑问等意图时，必须停留在当前节点，不能进入下一节点**

## 验收意图分类定义

### 验收通过意图（可进入下一节点）

| 意图类型 | 关键词示例 |
|---------|-----------|
| 明确确认 | "确认"、"没问题"、"可以"、"好的"、"通过"、"验收通过" |
| 推进指令 | "继续"、"下一步"、"往下走"、"进入下一阶段" |
| 满意表达 | "满意"、"就这样"、"定稿"、"OK"、"ok" |

### 非验收意图（停留在当前节点）

| 意图类型 | 关键词示例 | 处理方式 |
|---------|-----------|---------|
| 修改请求 | "修改"、"不对"、"重来"、"重新生成"、"调整"、"改一下" | 重新执行当前节点 |
| 补充需求 | "补充"、"还有"、"另外"、"加上"、"漏了" | 补充内容后重新展示 |
| 疑问咨询 | "为什么"、"什么意思"、"不懂"、"解释一下" | 解答疑问后继续等待确认 |
| 部分确认 | "这部分可以"、"其他的再改改" | 视为未完全确认，继续修改 |
| 拖延/犹豫 | "再看看"、"先这样"、"待定" | 明确询问用户是否确认 |
| 变更需求 | "变更"、"加功能"、"改需求"、"新增"、"删除" | 进入 change 节点 |

## 验收标准模板（每个节点通用）

每个节点增加的验收标准遵循以下统一模板：

```markdown
### X.X 用户验收标准

产出展示后，明确告知用户以下验收规则：

**验收通过意图（可进入下一节点）：**
- 明确确认："确认"、"没问题"、"可以"、"好的"、"通过"、"验收通过"
- 推进指令："继续"、"下一步"、"往下走"、"进入下一阶段"
- 满意表达："满意"、"就这样"、"定稿"、"OK"、"ok"

**非验收意图（停留在当前节点）：**
- 修改请求："修改"、"不对"、"重来"、"重新生成"、"调整"、"改一下" → 重新执行当前节点
- 补充需求："补充"、"还有"、"另外"、"加上"、"漏了" → 补充内容后重新展示
- 疑问咨询："为什么"、"什么意思"、"不懂"、"解释一下" → 解答疑问后继续等待确认
- 部分确认："这部分可以"、"其他的再改改" → 视为未完全确认，继续修改
- 拖延/犹豫："再看看"、"先这样"、"待定" → 明确询问用户是否确认

**重要规则：**
- 只有用户明确表达验收通过意图时，才能推进到下一节点
- 任何非验收意图（修改、补充、疑问、部分确认、犹豫）都必须停留在当前节点
- 用户表达"变更"、"加功能"、"改需求"等意图时，进入 change 节点
```
