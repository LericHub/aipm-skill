# 会员业务中台对接规范

## 1. 当时为什么这么决策

**背景**：组织内已经建设了会员信息中台，统一管理所有会员的基础信息、等级、积分等核心数据。

**核心原则**：
- 所有涉及会员等级、积分余额或其他会员信息的操作，都必须从会员信息中台获取
- 不得直连会员数据库，所有数据交互通过中台接口完成
- 中台已经对积分扣减的流程做了规范处理，需要遵循这套流程

**边界条件**：如果业务场景涉及非中台管理的会员数据（如临时数据、缓存数据），需要单独评估。

---

## 2. 通常落地方案是什么

### 2.1 核心接口清单

| 业务场景 | 接口编号 | 说明 |
|---------|---------|------|
| 查询会员等级和积分 | 3.2.2 | 查询会员基本信息 |
| 扣减会员积分 | 3.2.9 | 会员积分变更 |
| 查询积分变更结果 | 3.2.16 | 通过sequence_code查询变更是否成功 |
| 发放积分 | 3.2.17 | 上传消息，入参为行为积分编码 |

**注意**：接口详细参数请查阅接口文档3.2章节，文档中已定义各接口的请求参数和返回字段。

### 2.2 积分扣减标准流程

当业务需要扣减会员积分时，必须遵循以下流程：

```
1. 调用3.2.9接口扣减积分
2. 从3.2.9接口返回中获取sequence_code
3. 调用3.2.16接口，通过sequence_code查询是否扣减成功
4. 若查询到扣减成功 → 积分变更完成，结束流程
5. 若没有返回结果或超时 → 间隔2秒后重试3.2.16接口
6. 最多重试3次
7. 若3次重试后仍未查询到成功结果 → 认定此次积分变更失败
```

### 2.3 积分发放标准流程

当业务需要发放积分时：

1. 调用3.2.17接口（上传消息）
2. 入参传入行为积分编码，该编码中已指定发放的积分数量和类型
3. 无需额外查询，中台会根据行为积分编码自动处理

## 3. 类似场景怎么考虑

**适用检查点**：
- 是否涉及会员等级查询？
- 是否涉及积分余额查询？
- 是否需要扣减会员积分？
- 是否需要发放积分给会员？
- 涉及积分扣减时，是否已在活动配置阶段预置了行为积分编码？（参见「活动配置分层范式」）

**禁止事项**：
- 积分扣减后，在未确认3.2.16返回成功前，禁止标记积分变更完成
- 积分扣减和积分发放不得合并为同一业务事务，必须分别调用各自接口
- 禁止自行实现"先查余额再扣减"的乐观锁逻辑，必须通过中台3.2.9接口完成

**常见变体**：
- 积分扣减 + 积分发放组合：先扣后发，注意分别走各自的流程
- 积分扣减失败处理：需要回滚业务状态，并提示用户重试
- 批量积分操作：需要考虑接口调用频率限制，建议增加任务队列机制

**容易踩坑的地方**：
- 积分扣减后未查询结果就结束流程 → 必须通过3.2.16确认结果
- 重试间隔时间不对 → 必须间隔2秒，不是立即重试
- 重试次数超过限制 → 最多3次，不要无限重试

**积分变更失败的判断标准**：
- 3次重试后仍未查询到成功结果
- 接口返回明确错误码（如余额不足、会员不存在等）
