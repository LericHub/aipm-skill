# 页面装修与热区配置范式

## 1. 当时为什么这么决策

> 本质：回答"为什么这样判断"，传递隐性知识

### 业务背景和约束

互动营销小程序会面向多个品牌店铺使用，每个品牌的主题色、设计格调各异，需要在统一的小程序框架下支持灵活的页面装修能力。

同时，运营团队配置页面的学习成本和操作成本需要可控，不能做成一个需要大量素材准备和复杂配置理解的系统。

### 为什么不采用其他方案

- **不采用组件化低代码平台**：虽然理论上更灵活，但开发和运营成本高，第一版不需要这么强的扩展性。
- **不按组件维度拆分装修**：每拆出一个独立组件，就多一个需要配置和维护的素材项，容易造成运营困惑。
- **不追求可视化拖拽**：开发成本高，且运营实际只需要上传背景图+配置几个热区。

### 这个决策解决了什么核心矛盾

在**视觉效果灵活性**（支持多品牌主题）和**运营配置效率**（低学习成本、低素材成本）之间找到平衡。

核心思路：**用一张背景图承载整体视觉，把可变区域压缩到最少。**

### 边界条件

- 本范式适用于互动营销小程序。如用户未明确说明，优先按本方案设计装修。
- 如果品牌需要完全自定义皮肤（超出背景图能承载的范围），需要走定制开发。
- 本方案不适用于复杂交互动画、动态数据驱动的 UI。

## 2. 通常落地方案是什么

> 本质：回答"通常怎么落地"，提供可复用路径

### 典型页面组织：两层装修设计

```
┌─────────────────────────────────┐
│       小程序装修（骨架层）         │
│  配置小程序通用背景、按钮样式等      │
├─────────────────────────────────┤
│       活动装修（内容层）           │
│  配置活动特有的背景图、热区、卡券等   │
└─────────────────────────────────┘
```

- **小程序装修**：配置小程序骨架页面的样式，如首页背景、通用按钮样式、详情页背景。
- **活动装修**：配置随活动各异的装修内容，如活动发放的卡券装修等。

### 核心配置项

每个页面通常只需要以下配置项：

| 配置项 | 说明 |
|--------|------|
| 背景图 | 承载主要视觉表达，尺寸 750 × 1700 px |
| 热区列表 | 每个热区含坐标（按 HTML 设计稿尺寸）和跳转行为 |
| 局部文案（可选） | 需要运营编辑的少量文字 |
| 状态素材（可选） | 仅当按钮需要多状态时，如未激活/激活状态图 |

### 推荐的实现顺序

1. 先设计背景图，把按钮名称、输入项标题放在图上
2. 前端提供热区坐标（与背景图严格对应）
3. 后台配置热区位置和跳转行为
4. 运营只需上传背景图 + 少量热区配置

### 素材组织原则

- 优先减少单页素材数量，能用一张图就不用两张。
- 只有需要高频替换的局部元素，才单独抽离为可配置素材。
- 如果多个页面共用同一视觉资产，再考虑进入全局配置统一维护。

## 3. 类似场景怎么考虑

> 本质：回答"怎么迁移到新场景"，提供方法论

### 判断是否适用本方案的检查点

```
□ 页面差异主要来自视觉主题变化，而不是页面结构剧变？
□ 运营同学更需要低学习成本，而不是高灵活度？
□ 当前页面的交互复杂度在热区+跳转范围内？
```

如果以上全是 ✅，优先用背景图+热区方案。

如果 ❌ 较多，考虑：
- 页面结构本身变化大 → 需要组件化方案
- 需要动态数据展示 → 组件化或动态模板
- 需要复杂交互动画 → 走定制开发

### 可以直接复用的 vs 需要调整的

| 可以直接复用 | 需要调整 |
|-------------|---------|
| 背景图 + 热区的装修思路 | 背景图尺寸（不同设备可能不同） |
| 页面级配置的结构 | 热区坐标（按新设计稿重新标注） |
| 素材组织原则 | 配置后台的字段 |

### 常见变形

1. **按钮需要换状态**：拆成"背景图 + 未激活状态按钮图"两个配置项
2. **多品牌共用一套页面但主题色不同**：每品牌一套背景图，热区配置共用
3. **局部内容高频替换**：将该区域从背景图拆出，单独作为可配置素材

### 容易踩坑的地方

- 设计稿元素定位没有严格配合前端组件位置 → 热区对不准
- 背景图上放了太多可变元素 → 替换一次要重做整张图
- 热区数量过多 → 运营配置负担重，且后期维护困难
