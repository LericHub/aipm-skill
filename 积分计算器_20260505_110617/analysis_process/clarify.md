# 需求澄清

## 一、背景与痛点

> 已由 BRAINSTORM 阶段确认
>
> 用户原始回答：搭建一个积分计算器应用，天猫小程序，从0-1搭建

**归纳扩展**：
目前在会员运营场景中，用户购买商品前无法直观知道能获得多少积分，导致购买决策缺乏积分激励的透明度。需要一个面向 C 端会员的小程序工具，让用户能选商品、算积分。

## 二、业务目标

> 已由 BRAINSTORM 阶段确认
>
> 用户原始回答：帮助用户计算购买商品可获得的积分

**归纳扩展**：
通过透明化的积分试算，让用户"所见即所得"地知道买哪些商品能获得多少积分，从而激励用户完成购买行为。B 端管理端则支撑活动运营，配置推荐商品、监控活动效果。

## 三、上下游系统和系统边界

| 系统 | 提供能力 | 接口 |
|------|----------|------|
| 会员信息中台 | 积分查询、等级查询、积分试算 | DL-3.2.7 / DL-3.2.4 / DL 2.2.95 |
| 淘宝开放平台 | 批量获取商品详情 | taobao.items.seller.list.get |

**数据流**：C 端选中商品 → 传商品 ID 列表给后端 → 后端调会员中台积分试算接口 → 返回积分结果给前端展示。

## 四、接口文档

| 接口 | 文档 | 说明 |
|------|------|------|
| 会员中台 DL-3.2.7 / DL-3.2.4 / DL 2.2.95 | 暂无 | PRD 中仅标记数据来源 |
| 淘宝 items.seller.list.get | https://open.taobao.com/v2/doc#/apiFile?docId=24626&docType=2 | 批量获取商品详细信息 |

## 五、业务规则

- 积分计算规则由后端/中台处理，前端仅展示查询结果，不做本地计算
- 不同会员等级、不同活动的积分规则差异由后端和中台约定，前端不关心

## 六、业务风险

1. **商品时效性风险**：淘宝商品下架/改价后，管理端推荐列表仍是旧数据，导致 C 端点击跳转 404 或价格不准 → 管理端支持手动刷新/下架商品
2. **中台接口稳定性**：会员中台试算接口异常 → 前端 Toast 提示用户 + 管理端看板暴露异常
3. **活动与商品时间错配**：活动已过期但 C 端仍在展示 → 活动到期自动禁用

## 七、埋点场景

| 埋点项 | 说明 |

| 用户行为日志 | C 端接口异常场景（如中台调用失败），通过埋点报表可在管理端查看 |
